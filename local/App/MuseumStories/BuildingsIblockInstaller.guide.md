# Инструкция по использованию `BuildingsIblockInstaller.php`

## Что делает файл

`BuildingsIblockInstaller.php` содержит класс `App\MuseumStories\BuildingsIblockInstaller`.

Класс нужен для автоматического создания структуры инфоблока музейных зданий в Bitrix.

Он создаёт или проверяет:

- тип инфоблока;
- сам инфоблок зданий;
- набор пользовательских свойств инфоблока.

Метод `install()` идемпотентный: если инфоблок уже существует, файл не создаёт его повторно, а только проверяет и добавляет отсутствующие свойства.

## Что именно создаётся

Скрипт создаёт инфоблок по настройкам из `MuseumStoriesConfig`.

Из этого файла берутся:

- `IBLOCK_TYPE_ID` — код типа инфоблока;
- `IBLOCK_CODE` — код инфоблока;
- `IBLOCK_API_CODE` — API-код инфоблока;
- `IBLOCK_NAME` — название инфоблока;
- `IBLOCK_DESCRIPTION` — описание инфоблока;
- `DEFAULT_SITE_IDS` — сайты, к которым привязывается инфоблок;
- коды свойств инфоблока.

Пользовательские свойства, которые создаёт класс:

| Название | Тип | Назначение |
|---|---:|---|
| Источник | число | ID внешнего источника данных |
| Адрес | строка | адрес здания |
| Координаты на карте | строка | координаты для карты |
| Закрыто | список | флаг закрытия здания: `Да` / `Нет` |
| Ссылка на билеты | строка | URL для покупки билетов |
| Путь к картинке | строка | путь к изображению |
| Таймлайн | строка | исторический таймлайн |

## Что нужно перед запуском

### 1. Рабочее ядро Bitrix

Файл должен запускаться внутри установленного Bitrix.

Должен существовать путь:

```text
/bitrix/modules/main/include/prolog_before.php
```

Сам класс не подключает `prolog_before.php`. Это должен сделать внешний запускной файл.

### 2. Модуль `iblock`

Нужен установленный модуль инфоблоков.

Внутри класса модуль подключается так:

```php
Loader::includeModule('iblock')
```

Если модуль не подключится, будет выброшена ошибка `SystemException`.

### 3. Класс `MuseumStoriesConfig`

Файл зависит от класса:

```php
App\MuseumStories\MuseumStoriesConfig
```

В нём должны быть объявлены все константы, которые использует установщик.

Минимально нужны такие константы:

```php
IBLOCK_TYPE_ID
IBLOCK_CODE
IBLOCK_API_CODE
IBLOCK_NAME
IBLOCK_DESCRIPTION
DEFAULT_SITE_IDS
PROPERTY_SOURCE_ID
PROPERTY_ADDRESS_RU
PROPERTY_MAP_COORDS
PROPERTY_CLOSED_FLAG
PROPERTY_TICKET_URL
PROPERTY_PICTURE_PATH
PROPERTY_TIMELINE_RU
```

### 4. Автозагрузка классов

Классы проекта должны быть доступны через Composer autoload или другой автозагрузчик Bitrix.

Например, если используется Composer, во внешнем запускном файле должен быть подключён:

```php
/local/vendor/autoload.php
```

### 5. Права на создание инфоблоков

Пользователь или процесс, от имени которого запускается код, должен иметь права на:

- создание типов инфоблоков;
- создание инфоблоков;
- создание свойств инфоблоков;
- запись в базу данных.

### 6. Входные данные

Метод `install()` принимает необязательный массив кодов сайтов:

```php
$installer->install(['s1']);
```

Если массив пустой, будут использованы сайты из:

```php
MuseumStoriesConfig::DEFAULT_SITE_IDS
```

## Как запустить

Сам файл — это класс, а не самостоятельный скрипт.

Его нужно вызвать из отдельного PHP-файла, например:

```text
/local/scripts/install-buildings-iblock.php
```

### Пример 1. Запуск через браузер

Допущение: создан файл `/local/scripts/install-buildings-iblock.php`, который подключает ядро Bitrix, автозагрузку и вызывает `BuildingsIblockInstaller`.

Открыть в браузере:

```text
https://example.ru/local/scripts/install-buildings-iblock.php
```

### Пример 2. Запуск из CLI внутри контейнера

Допущение: проект находится в Docker-контейнере Bitrix, а сайт лежит в `/opt/www`.

```bash
php /opt/www/local/scripts/install-buildings-iblock.php
```

### Пример 3. Запуск через Docker

Допущение: PHP-контейнер называется `dev_php`.

```bash
docker exec -it dev_php php /opt/www/local/scripts/install-buildings-iblock.php
```

## Пример внешнего запускного файла

Это не содержимое `BuildingsIblockInstaller.php`, а отдельный файл-обёртка.

```php
<?php

$_SERVER['DOCUMENT_ROOT'] = $_SERVER['DOCUMENT_ROOT'] ?: '/opt/www';

require_once $_SERVER['DOCUMENT_ROOT'] . '/bitrix/modules/main/include/prolog_before.php';
require_once $_SERVER['DOCUMENT_ROOT'] . '/local/vendor/autoload.php';

use App\MuseumStories\BuildingsIblockInstaller;

$installer = new BuildingsIblockInstaller();
$result = $installer->install(['s1']);

if ($result['success']) {
    echo 'Инфоблок создан или уже существует. ID: ' . $result['id'] . PHP_EOL;
    return;
}

echo 'Ошибка установки:' . PHP_EOL;
print_r($result['errors']);
```

## Основные параметры

### `install(array $siteIds = [])`

Главный публичный метод.

Создаёт тип инфоблока, инфоблок и свойства.

Параметр:

| Параметр | Тип | Описание |
|---|---|---|
| `$siteIds` | `array` | Список сайтов Bitrix, к которым нужно привязать инфоблок |

Пример:

```php
$result = $installer->install(['s1']);
```

Если передать пустой массив:

```php
$result = $installer->install();
```

будет использовано значение из:

```php
MuseumStoriesConfig::DEFAULT_SITE_IDS
```

## Что возвращает `install()`

Метод возвращает массив:

```php
[
    'success' => true,
    'id' => 25,
    'errors' => [],
]
```

Описание полей:

| Поле | Тип | Значение |
|---|---|---|
| `success` | `bool` | Успешно ли прошла установка |
| `id` | `int|null` | ID инфоблока, если он создан или найден |
| `errors` | `array` | Список ошибок |

## Как работает логика установки

1. Подключается модуль `iblock`.
2. Нормализуется список сайтов.
3. Проверяется наличие типа инфоблока.
4. Если типа нет — создаётся новый тип инфоблока.
5. Проверяется наличие инфоблока по коду.
6. Если инфоблок найден в другом типе — возвращается ошибка.
7. Если инфоблока нет — создаётся новый инфоблок.
8. Проверяются свойства инфоблока.
9. Отсутствующие свойства создаются.
10. Если всё успешно — транзакция фиксируется через `Commit()`.
11. Если есть ошибка — транзакция откатывается через `Rollback()`.

## Важные особенности

### Инфоблок создаётся в режиме `VERSION = 2`

Это значит, что значения свойств будут храниться в отдельных таблицах инфоблока.

Для Bitrix это нормальный и современный вариант работы со свойствами.

### Повторный запуск безопасен

Файл можно запускать несколько раз.

Если инфоблок уже создан, повторно он не создаётся.

Если часть свойств уже есть, они пропускаются.

### Элементы инфоблока не создаются

Этот файл создаёт только структуру:

- тип инфоблока;
- инфоблок;
- свойства.

Он не создаёт элементы внутри инфоблока.

Для создания элементов нужен отдельный код через `CIBlockElement::Add()` или D7 ORM для инфоблоков.

## Маленькие примеры использования

```php
$installer = new BuildingsIblockInstaller();
$result = $installer->install(['s1']);
```

### Создать инфоблок для нескольких сайтов

```php
$result = $installer->install(['s1', 's2']);
```

```php
$result = $installer->install();
```

```php
if (!$result['success']) {

}
```

## FAQ

### Вопрос → Этот файл создаёт элементы инфоблока?

Короткий ответ → Нет. Он создаёт только структуру инфоблока и свойства.

### Вопрос → Можно ли запускать файл повторно?

Короткий ответ → Да. Метод `install()` проверяет наличие инфоблока и свойств перед созданием.

### Вопрос → Где задаётся код инфоблока?

Короткий ответ → В `MuseumStoriesConfig::IBLOCK_CODE`.

### Вопрос → Где задаются коды свойств?

Короткий ответ → В константах `MuseumStoriesConfig::PROPERTY_*`.

### Вопрос → Можно ли изменить список свойств?

Короткий ответ → Да. Для этого нужно изменить метод `getPropertyDefinitions()`.

### Вопрос → Можно ли создать инфоблок без свойств?

Короткий ответ → Да. Нужно убрать или изменить вызов `ensureProperties()`.

### Вопрос → Почему используется `CIBlock::Add`, а не ORM `DataManager`?

Короткий ответ → Потому что инфоблок создаётся через API модуля `iblock`. ORM-модель используется для работы с данными, а не для первичного создания структуры инфоблока.

### Вопрос → Почему часть кода использует D7, а часть старые классы `CIBlock`?

Короткий ответ → В Bitrix это обычная практика: проверку можно делать через D7-таблицы, а создание инфоблока и свойств часто выполняется через классический API `CIBlock`, `CIBlockType`, `CIBlockProperty`.

## Troubleshooting

### Ошибка: `Class "App\MuseumStories\MuseumStoriesConfig" not found`

Причина → Не подключён автозагрузчик или нет файла с классом `MuseumStoriesConfig`.

Как исправить:

- проверь namespace класса;
- проверь Composer autoload;
- выполни `composer dump-autoload`;
- проверь подключение `/local/vendor/autoload.php`.

### Ошибка: `Модуль iblock не подключен`

Причина → Модуль инфоблоков не установлен или не подключается.

Как исправить:

- проверь, установлен ли модуль `iblock` в админке Bitrix;
- проверь, что код запускается внутри Bitrix;
- проверь подключение `prolog_before.php`.

### Ошибка: `Call to undefined function` или не найдены классы Bitrix

Причина → Скрипт запущен без ядра Bitrix.

Как исправить:

- подключи `prolog_before.php` перед вызовом установщика;
- при CLI-запуске явно задай `$_SERVER['DOCUMENT_ROOT']`.

### Ошибка: инфоблок уже существует в другом типе

Причина → В базе уже есть инфоблок с таким же `CODE`, но с другим `IBLOCK_TYPE_ID`.

Как исправить:

- измени `MuseumStoriesConfig::IBLOCK_CODE`;
- или перенеси существующий инфоблок в нужный тип;
- или удали конфликтующий инфоблок, если он не нужен.

### Свойство не создаётся

Причина → Ошибка в описании свойства, например неверный `PROPERTY_TYPE`, пустой `CODE` или конфликт кода.

Как исправить:

- проверь массив в `getPropertyDefinitions()`;
- проверь `LAST_ERROR`, который возвращает Bitrix;
- проверь, нет ли свойства с таким кодом в инфоблоке.

### Инфоблок создался, но не виден в нужном сайте

Причина → Неверный список `SITE_ID`.

Как исправить:

- передай правильный сайт в `install(['s1'])`;
- проверь `MuseumStoriesConfig::DEFAULT_SITE_IDS`;
- проверь существующие сайты в админке Bitrix.

### При CLI-запуске пустой `DOCUMENT_ROOT`

Причина → В CLI переменная `$_SERVER['DOCUMENT_ROOT']` часто не заполнена.

Как исправить:

Перед подключением ядра задай путь вручную:

```php
$_SERVER['DOCUMENT_ROOT'] = '/opt/www';
```

Для Timeweb или другого хостинга путь будет другим.

## Рекомендации по использованию

- Не запускай установщик на продакшене без резервной копии базы.
- После изменения списка свойств запускай файл повторно: он добавит только отсутствующие свойства.
- Не меняй коды свойств после того, как на них уже завязан рабочий код.
- Для удаления инфоблока или свойств лучше сделать отдельный uninstall-скрипт.
- Для учебного проекта полезно держать такой установщик рядом с конфигом и сервисами модуля.
